# CLAUDE.md

이 파일은 Claude Code(및 협업 개발자)가 이 프로젝트에서 작업할 때 참고하는 가이드 문서입니다.

## 프로젝트 개요

한국대학신문(news.unn.net)에서 "AIDX", "AID", "AI" 관련 기사를 수집·분류·요약하여 보여주는 웹 대시보드입니다.

### 핵심 기능

- **수집**: 한국대학신문 RSS(`https://news.unn.net/rssIndex.html`) 또는 기사 목록 페이지를 크롤링하여 신규 기사를 가져온다.
- **필터링**: 아래 키워드가 제목 또는 본문에 포함된 기사만 대상으로 한다.
  - `AIDX`, `AID`, `AI`, `인공지능`, `디지털전환`, `디지털 전환`
- **저장**: 수집된 기사를 DB(Supabase)에 저장한다. 원문 URL, 제목, 발행일, 본문(또는 요약 대상 텍스트) 등을 보관한다.
- **요약/분류**: LLM API(현재 Gemini API — 무료 티어 사용을 위해 채택, `GEMINI_API_KEY` 필요)를 이용해 각 기사에 3줄 요약과 카테고리를 생성하여 DB에 함께 저장한다. 카드 등록이 가능해지면 Claude API로 교체할 수 있도록 `lib/summarize/` 안에서 클라이언트만 분리되어 있다.
- **표시**: 웹 대시보드(Next.js)에서 카테고리별 / 날짜별로 기사를 조회·필터링할 수 있게 보여준다.
- **스케줄링**: GitHub Actions로 주기적으로 크롤링 파이프라인을 실행한다(신규 기사 수집 → 필터링 → 저장 → 요약/분류).

### 데이터 흐름 (개략)

```
[한국대학신문 RSS/목록 페이지]
        │  크롤링 (GitHub Actions 스케줄)
        ▼
   키워드 필터링
        │
        ▼
   Supabase DB 저장 (원문)
        │
        ▼
   Gemini API 요약/카테고리 생성
        │
        ▼
   Supabase DB 업데이트 (요약, 카테고리)
        │
        ▼
   Next.js 대시보드에서 조회 (카테고리별 / 날짜별)
```

## 기술 스택

| 영역 | 기술 |
|---|---|
| 프론트엔드 + API | Next.js (App Router 권장) |
| 데이터베이스 | Supabase (Postgres) |
| 요약/분류 | Gemini API (무료 티어, `lib/summarize/`) — 카드 등록 시 Claude API로 교체 가능 |
| 스케줄링 크롤링 | GitHub Actions |
| 언어 | TypeScript |

## 폴더 구조 원칙

아직 코드는 작성하지 않은 초기 단계이며, 아래는 이후 구현 시 따를 구조 원칙입니다. 실제 폴더는 구현 시점에 필요에 맞게 생성합니다.

- **`app/`**: Next.js App Router 기반 페이지 및 API 라우트.
  - `app/(dashboard)/`: 대시보드 UI 페이지 (카테고리별/날짜별 뷰 등).
  - `app/api/`: 서버 API 라우트 (필요 시 내부 트리거, 웹훅 등).
- **`lib/`**: 도메인 로직과 외부 연동 코드를 기능 단위로 분리.
  - `lib/crawler/`: RSS/페이지 크롤링 및 파싱 로직.
  - `lib/filter/`: 키워드 필터링 로직.
  - `lib/summarize/`: LLM API(현재 Gemini) 호출 및 요약/카테고리 생성 로직.
  - `lib/db/`: Supabase 클라이언트 및 쿼리 함수.
  - `lib/types/`: 프로젝트 전역에서 쓰는 타입 정의.
- **`components/`**: 재사용 가능한 UI 함수형 컴포넌트. 페이지 전용 컴포넌트는 해당 페이지 폴더 내부에 둔다.
- **`scripts/`**: GitHub Actions 등에서 실행되는 배치성 스크립트(크롤링 트리거, 백필 등).
- **`.github/workflows/`**: GitHub Actions 워크플로 정의 (크롤링 스케줄링).
- **`supabase/`**: 스키마 마이그레이션, DB 관련 설정.
- **`tests/`** (또는 각 모듈과 동일 위치의 `*.test.ts`): 단위/통합 테스트.

폴더 구조 원칙:

1. **관심사 분리**: 크롤링, 필터링, 요약, DB 접근, UI 렌더링을 각각 독립된 모듈로 분리하여 하나의 파일/폴더가 여러 책임을 갖지 않도록 한다.
2. **UI와 로직 분리**: 컴포넌트(`components/`, `app/`)는 표시에 집중하고, 데이터 수집·가공 로직은 `lib/`에 둔다.
3. **외부 연동은 어댑터로 감싼다**: 한국대학신문 크롤링, Supabase, LLM API(Gemini/Claude) 호출은 각각 별도 모듈로 감싸서 교체·모킹이 쉽도록 한다.
4. **스크립트와 앱 코드 분리**: GitHub Actions에서 실행되는 배치 스크립트(`scripts/`)와 Next.js 앱 코드(`app/`, `lib/`, `components/`)를 명확히 구분한다.

## 코딩 컨벤션

- **언어**: 모든 코드는 TypeScript로 작성한다. `any` 타입 사용을 지양하고, 외부 데이터(크롤링 결과, API 응답, DB 로우)는 명시적으로 타입/스키마를 정의한다.
- **컴포넌트**: React 컴포넌트는 함수형 컴포넌트로 작성한다. 클래스형 컴포넌트는 사용하지 않는다.
- **에러 핸들링**: 모든 외부 I/O(크롤링 요청, Supabase 쿼리, LLM API 호출)는 반드시 에러 핸들링을 포함한다.
  - 실패 시 전체 파이프라인이 죽지 않도록 개별 기사 단위로 실패를 격리하고 로그를 남긴다.
  - 크롤링/요약 실패는 재시도 또는 스킵 후 다음 항목으로 진행하는 방식을 기본으로 한다.
  - API 라우트에서는 예외를 잡아 적절한 HTTP 상태 코드와 에러 메시지를 반환한다.
- **네이밍**: 함수/변수는 의미가 드러나는 이름을 사용한다 (예: `filterArticlesByKeyword`, `summarizeArticle`).
- **비동기 처리**: `async/await`를 사용하고, `Promise` 체이닝은 지양한다.
- **환경 변수**: Supabase, Gemini API 등의 키/시크릿은 코드에 하드코딩하지 않고 환경 변수(`.env` 등)로 관리한다.
- **커밋 단위**: 크롤링, 필터링, 요약, 대시보드 UI 등 기능 단위로 작게 커밋한다.

## 참고 사항

- 이 문서는 프로젝트 초기 설계 단계에서 작성되었으며, 아직 실제 코드는 구현되지 않았다.
- 구현을 시작할 때는 이 문서의 원칙(폴더 구조, 관심사 분리, 에러 핸들링)을 기준으로 진행한다.
